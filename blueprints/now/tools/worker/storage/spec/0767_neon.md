# 0767 — Neon Serverless PostgreSQL Driver

Architecture spec for the `NeonEngine` storage driver (`src/storage/neon_driver.ts`).
This driver connects Cloudflare Workers to Neon Serverless PostgreSQL using the
`@neondatabase/serverless` package — no TCP sockets required.

---

## 1. Overview

Neon is a serverless PostgreSQL provider that exposes two non-TCP transports:

1. **HTTP query endpoint** — send a single SQL statement as an HTTP POST, get
   rows back as JSON. Zero connection overhead. One round trip per query.
2. **WebSocket endpoint** — full PostgreSQL wire protocol tunneled over a
   WebSocket. Supports interactive transactions (BEGIN/COMMIT), prepared
   statements, and multi-statement sessions.

Cloudflare Workers cannot open TCP sockets to arbitrary hosts, but they can
make HTTP requests and open WebSockets. Neon's transport layer slots in
natively without Hyperdrive or any other proxy.

The `NeonEngine` class extends `PgEngineBase` (shared with `HyperdriveEngine`)
and implements two abstract methods:

- `query()` — uses HTTP mode for single-statement reads.
- `transaction()` — uses WebSocket Pool for multi-statement write transactions.

All `StorageEngine` interface methods (write, read, list, search, head, stats,
move, delete, log, presign) are implemented in `PgEngineBase`. The Neon driver
only provides the transport.

## 2. Architecture

```
                          ┌─────────────────────────┐
                          │     Neon SQL Proxy       │
  ┌──────────┐   HTTP     │  (stateless, edge PoP)  │     ┌──────────────┐
  │          │───────────►│                         │────►│              │
  │          │  POST /sql │  Parse SQL, forward to  │     │  PostgreSQL  │
  │          │◄───────────│  compute, return JSON   │◄────│  (compute)   │
  │          │   JSON     └─────────────────────────┘     │              │
  │  Worker  │                                            │  Neon stores │
  │          │            ┌─────────────────────────┐     │  data on S3  │
  │          │   WS       │    Neon WS Proxy        │     │  (pageserver)│
  │          │───────────►│  (connection upgrade)   │────►│              │
  │          │  PG wire   │  Full PG wire protocol  │◄────│              │
  │          │◄───────────│  over WebSocket frames  │     └──────────────┘
  │          │            └─────────────────────────┘
  │          │
  │          │            ┌─────────────────────────┐
  │          │   HTTP     │                         │
  │          │───────────►│     R2 Bucket           │
  │          │◄───────────│  (blob storage, S3 API) │
  └──────────┘            └─────────────────────────┘

  ─── Read path (HTTP):  Worker → Neon SQL Proxy → PG → JSON response
  ─── Write path (WS):   Worker → Neon WS Proxy → PG (BEGIN/COMMIT)
  ─── Blob path (HTTP):  Worker → R2 (PUT/GET content-addressed blobs)
```

### Transport selection

| Operation type     | Transport      | Why                                           |
|--------------------|----------------|-----------------------------------------------|
| Single-row reads   | HTTP `neon()`  | Zero overhead, one round trip, stateless      |
| Multi-row reads    | HTTP `neon()`  | Same — every query is independent             |
| Write transactions | WebSocket Pool | Need BEGIN/COMMIT across multiple statements  |
| Schema DDL         | HTTP `neon()`  | Single `CREATE TABLE IF NOT EXISTS` per call  |

## 3. Neon SQL Proxy — HTTP Mode

The `@neondatabase/serverless` package's `neon()` function returns an HTTP-based
SQL executor. Each call:

1. Serializes the SQL + parameters as JSON.
2. Sends a single `POST` to `https://{project}.neon.tech/sql`.
3. Neon's edge proxy parses the request, routes it to the correct compute
   endpoint, executes the query, and returns rows as JSON.
4. The package deserializes the response into JavaScript objects.

```typescript
import { neon } from "@neondatabase/serverless";

const sql = neon(connectionString);
const rows = await sql.query("SELECT * FROM stg_files WHERE owner = $1", [actor]);
```

Key properties:

- **Stateless.** No connection is opened or pooled. Each query is an
  independent HTTP request. There is no connection startup time (no TLS
  handshake, no PG auth, no parameter negotiation).
- **One round trip.** Exactly one HTTP request/response. Latency is dominated
  by network distance to the nearest Neon SQL proxy edge PoP, plus compute
  execution time.
- **No transaction support.** HTTP mode cannot span multiple queries in a
  transaction. Each query auto-commits.
- **No prepared statements.** The HTTP endpoint takes SQL text + parameters
  directly. Neon does server-side parameter binding.

This is ideal for all read operations (head, list, search, stats, log) because
they are single-query, read-only operations.

## 4. WebSocket Pool — Transaction Mode

For write operations that require atomicity across multiple SQL statements
(INSERT into events + UPSERT into files + UPSERT into blobs), the driver uses
`Pool` from `@neondatabase/serverless`.

```typescript
import { Pool } from "@neondatabase/serverless";

const pool = new Pool({ connectionString: poolDsn });
const client = await pool.connect();
await client.query("BEGIN");
// ... multiple queries ...
await client.query("COMMIT");
client.release();
```

The Pool opens a WebSocket to Neon's WebSocket proxy, which tunnels the full
PostgreSQL wire protocol over WebSocket frames. This gives:

- **Interactive transactions.** BEGIN/COMMIT/ROLLBACK with full ACID semantics.
- **Multiple queries per connection.** The client stays connected for the
  duration of the transaction.
- **Connection pooling.** `Pool` reuses WebSocket connections across requests
  within the same Worker isolate lifetime.

### DSN sanitization

Neon DSN strings sometimes include `channel_binding=require` for TLS channel
binding. This is not supported over the WebSocket proxy (which terminates TLS
at the proxy layer, not end-to-end). The driver strips this parameter:

```typescript
const poolDsn = config.connectionString
  .replace(/([?&])channel_binding=[^&]*/g, "$1")
  .replace(/[?&]$/, "");
```

### Write transaction anatomy

A typical `write()` call executes 5-6 SQL statements inside a single
transaction:

```
BEGIN
  SELECT addr FROM stg_files WHERE owner = $1 AND path = $2     -- check existing
  INSERT INTO stg_tx ... RETURNING next_tx                       -- allocate tx number
  INSERT INTO stg_events (tx, actor, action, path, ...)          -- record event
  INSERT INTO stg_files ... ON CONFLICT DO UPDATE                -- upsert file record
  INSERT INTO stg_blobs ... ON CONFLICT DO UPDATE                -- upsert blob ref
  UPDATE stg_blobs SET ref_count = ref_count - 1 ...             -- decrement old blob (if overwrite)
COMMIT
```

Each statement is a separate WebSocket message. The total latency is:

```
WebSocket upgrade (~50ms first time, reused after) +
N * (send frame + PG execute + return frame) +
COMMIT flush
```

This explains why writes average ~310ms while reads average ~52ms.

## 5. PostgreSQL Schema

The schema is defined in `PgEngineBase` and shared with `HyperdriveEngine`.
Tables use the `stg_` prefix to avoid conflicts in shared databases.

```sql
-- File metadata (materialized view of current state)
CREATE TABLE IF NOT EXISTS stg_files (
  owner      TEXT    NOT NULL,
  path       TEXT    NOT NULL,
  name       TEXT    NOT NULL,
  size       BIGINT  NOT NULL DEFAULT 0,
  type       TEXT    NOT NULL DEFAULT 'application/octet-stream',
  addr       TEXT,                          -- SHA-256 content address
  tx         INTEGER,                       -- last-write transaction number
  tx_time    BIGINT,                        -- timestamp of last-write tx
  updated_at BIGINT  NOT NULL,
  PRIMARY KEY (owner, path)
);
CREATE INDEX IF NOT EXISTS idx_stg_files_name    ON stg_files(owner, lower(name));
CREATE INDEX IF NOT EXISTS idx_stg_files_updated ON stg_files(owner, updated_at DESC);

-- Event log (append-only mutation history)
CREATE TABLE IF NOT EXISTS stg_events (
  id     BIGSERIAL PRIMARY KEY,
  tx     INTEGER   NOT NULL,
  actor  TEXT      NOT NULL,
  action TEXT      NOT NULL CHECK(action IN ('write','move','delete')),
  path   TEXT      NOT NULL,
  addr   TEXT,
  size   BIGINT    NOT NULL DEFAULT 0,
  type   TEXT,
  meta   TEXT,                              -- JSON: {from?} for moves
  msg    TEXT,                              -- commit message
  ts     BIGINT    NOT NULL
);
CREATE INDEX IF NOT EXISTS idx_stg_events_actor_tx ON stg_events(actor, tx DESC);
CREATE INDEX IF NOT EXISTS idx_stg_events_path     ON stg_events(actor, path, tx DESC);

-- Blob reference tracking (for GC)
CREATE TABLE IF NOT EXISTS stg_blobs (
  addr       TEXT    NOT NULL,
  actor      TEXT    NOT NULL,
  size       BIGINT  NOT NULL,
  ref_count  INTEGER NOT NULL DEFAULT 1,
  created_at BIGINT  NOT NULL,
  PRIMARY KEY (addr, actor)
);

-- Per-actor transaction counter
CREATE TABLE IF NOT EXISTS stg_tx (
  actor   TEXT    PRIMARY KEY,
  next_tx INTEGER NOT NULL DEFAULT 1
);
```

### Schema initialization

Schema DDL is executed lazily on first request per Worker isolate. A
module-level `schemaReady` flag prevents re-execution. All statements use
`CREATE TABLE IF NOT EXISTS` / `CREATE INDEX IF NOT EXISTS` for idempotency.

### Differences from D1 driver

| Aspect              | D1 (SQLite)                          | Neon (PostgreSQL)                    |
|---------------------|--------------------------------------|--------------------------------------|
| Table scoping       | Per-actor sharded (`f_{shard}`)      | Shared tables with `owner` column    |
| Primary keys        | Composite TEXT keys                  | Same, but PG handles them natively   |
| Auto-increment      | `AUTOINCREMENT`                      | `BIGSERIAL`                          |
| Transactions        | `db.batch()` (SQLite batch)          | `BEGIN`/`COMMIT` over WebSocket      |
| Case-insensitive    | `COLLATE NOCASE`                     | `ILIKE` / `lower()` in index        |
| Index on function   | Not supported                        | `lower(name)` functional index       |
| Sequence            | Managed in `meta` table              | `stg_tx` table with `RETURNING`      |

## 6. Benchmark Results

Measured against a real Neon database in `ap-southeast-1` from a local
development machine. Tests use `@cloudflare/vitest-pool-workers` with
Miniflare for the R2 bucket but hit the remote Neon database over the
internet for SQL.

10 iterations for writes, 15 for reads. 2 warmup iterations excluded.

### Latency

| Operation | Avg (ms) | p50 (ms) | p95 (ms) | ops/sec |
|-----------|----------|----------|----------|---------|
| write     | 311      | 312      | 316      | 3.2     |
| head      | 52       | 52       | 54       | 19.3    |
| list      | 51       | 51       | 53       | 19.5    |
| search    | 53       | 52       | 63       | 19.0    |
| stats     | 52       | 52       | 54       | 19.2    |
| delete    | 362      | 363      | 384      | 2.8     |
| move      | 359      | 361      | 362      | 2.8     |

### Analysis

**Reads (~52ms)** use HTTP mode. The latency breaks down as:

```
~5ms   local network → Neon edge PoP
~2ms   Neon proxy parse + route
~40ms  Neon compute startup (cold) or query execution (warm)
~5ms   response serialization + return
```

Neon compute auto-suspends after 5 minutes of inactivity. Cold start adds
~200-300ms on the first query after suspension. The warmup iterations in the
benchmark absorb this. In production with steady traffic, compute stays warm
and reads should be ~30-50ms.

**Writes (~310-360ms)** use WebSocket mode. The latency breaks down as:

```
~50ms  WebSocket upgrade (first connection in pool)
~5-6 queries * ~45ms each = ~250ms  (serial, over WebSocket)
~10ms  COMMIT + pool return
```

Write transactions are inherently serial — each SQL statement depends on the
previous one (tx allocation depends on actor, event insertion depends on tx,
blob ref depends on addr). The WebSocket round-trip per statement dominates.

**Delete and move (~360ms)** are slightly slower than write because they
include an extra SELECT to read the existing file before the mutation.

### Comparison with D1 and DO (Miniflare, local)

These numbers are NOT directly comparable (different environments), but show
the relative positioning:

| Operation | D1 local (ms) | DO local (ms) | Neon remote (ms) |
|-----------|---------------|---------------|-------------------|
| write     | 1.5           | 1.1           | 311               |
| head      | 0.17          | 0.28          | 52                |
| list      | 0.26          | 0.40          | 51                |
| delete    | 0.57          | 0.37          | 362               |

Local D1/DO numbers are artificially fast (no network). In production,
D1 writes take ~10-50ms, DO writes ~5-30ms. Neon writes will remain
~300ms+ due to the multi-round-trip WebSocket transaction model.

For reads, production D1 edge replicas serve in ~1-5ms, while Neon HTTP
mode is ~30-50ms. The gap is real: D1 reads from local SQLite replicas
at the edge, Neon reads require a round trip to the database region.

## 8. Configuration

### Environment variables

```toml
# wrangler.toml
[vars]
STORAGE_DRIVER = "neon"

# Secrets (wrangler secret put)
# POSTGRES_DSN = "postgresql://user:pass@ep-xxx.ap-southeast-1.aws.neon.tech/dbname?sslmode=require"
```

The DSN must be a Neon connection string. The driver derives two connections
from this single DSN:

1. **HTTP mode** — passes the DSN directly to `neon()`. The package extracts
   the project endpoint from the hostname and uses the HTTP SQL API.
2. **WebSocket Pool** — passes the DSN (with `channel_binding` stripped) to
   `new Pool()`. The package opens a WebSocket to the same endpoint.

### TypeScript types

```typescript
// src/types.ts
interface Env {
  STORAGE_DRIVER?: string;   // "d1" | "do" | "hyperdrive" | "neon"
  POSTGRES_DSN?: string;     // Neon connection string
  BUCKET: R2Bucket;          // R2 binding (required for all drivers)
  // ... other env vars
}
```

### Driver selection

```typescript
// src/index.ts — engine middleware
const driver = c.env.STORAGE_DRIVER;
const engine =
  driver === "do" && c.env.STORAGE_DO
    ? new DOEngine({ ns: c.env.STORAGE_DO!, ...r2Config })
    : driver === "hyperdrive" && c.env.HYPERDRIVE
      ? new HyperdriveEngine({ connectionString: c.env.HYPERDRIVE.connectionString, ...r2Config })
      : driver === "neon" && c.env.POSTGRES_DSN
        ? new NeonEngine({ connectionString: c.env.POSTGRES_DSN, ...r2Config })
        : new D1Engine({ db: c.env.DB, ...r2Config });
```

### Testing

The Neon driver is tested against a real Neon database. Tests require
`POSTGRES_DSN` in the Vitest config bindings (loaded from `~/.local.env`).
Tests skip gracefully if the DSN is not configured.

```bash
# Run Neon-specific tests
npx vitest run src/__tests__/pg.test.ts
npx vitest run src/__tests__/pg_benchmark.test.ts
```

## 9. When to Use

### Choose Neon when:

- **You want a real PostgreSQL database.** Full SQL, joins, CTEs, JSON
  operators, extensions. Not SQLite (D1) or per-actor isolated stores (DO).
- **You need cross-actor queries.** All actors share one PostgreSQL database
  with standard `WHERE owner = $1` filtering. Analytics, admin dashboards,
  and cross-actor reporting are trivial.
- **You want cheap storage.** At $0.12/GiB/month, Neon is 6x cheaper than
  D1 ($0.75/GB) for storage-heavy workloads.
- **You already have a Neon database.** Adding storage tables to an existing
  Neon project is zero incremental infrastructure.
- **You need branching.** Neon supports database branching (copy-on-write
  clones). Useful for staging environments, testing migrations, or preview
  deployments.
- **You want to avoid Cloudflare lock-in.** The PostgreSQL schema works with
  any PostgreSQL host. Switching from Neon to Supabase or RDS requires only
  changing the DSN (and switching to the Hyperdrive driver for TCP-based
  providers).

### Do NOT choose Neon when:

- **Read latency is critical.** D1 edge replicas serve reads in ~1-5ms. Neon
  HTTP mode is ~30-50ms minimum. If your workload is read-heavy and latency-
  sensitive, D1 is significantly faster.
- **Write latency is critical.** WebSocket transaction mode adds ~300ms per
  write. D1 writes to primary in ~10-50ms. DO writes are ~5-30ms.
- **You need per-actor isolation.** Neon uses shared tables. DO provides
  process-level isolation (separate SQLite per actor).
- **You want zero external dependencies.** D1 and DO are built into Cloudflare.
  Neon is a third-party service with its own uptime SLA.
- **Cost is the primary concern for compute-light workloads.** D1's pricing
  model (rows read/written) is cheaper than Neon's compute-hour model for
  workloads under ~10K operations/day.

## 10. Trade-offs vs D1 / DO / Hyperdrive

```
                    D1Engine        DOEngine        NeonEngine      HyperdriveEngine
                    ────────        ────────        ──────────      ────────────────
Database:           Shared SQLite   Per-actor       Shared PG       Shared PG
                                    SQLite          (Neon)          (any PG host)

Read latency:       ~1-5ms          ~10-30ms        ~50ms           ~10-30ms
                    (edge replica)  (DO region)     (HTTP to Neon)  (Hyperdrive cache)

Write latency:      ~10-50ms        ~5-30ms         ~310ms          ~50-100ms
                    (to primary)    (local SQLite)  (WS txn)        (TCP via proxy)

Write throughput:   ~100-500/sec    Unlimited       ~3/sec          ~10-20/sec
                    (global)        (per-actor)     (serial WS)     (per connection)

Cost per SQL op:    ~$0.000001      ~$0.00015       ~$0.0001        ~$0.000001
                    (rows read)     (DO request)    (compute hour)  (rows + Hyperdrive)

Storage per GB:     $0.75           $0.20           $0.12           $0.12+
                                    (DO SQLite)     (Neon)          (varies by host)

Max DB size:        10 GB           1 GB/actor      Unlimited       Unlimited
                    (shared)        (DO limit)      (Neon Pro)      (depends on host)

Cross-actor query:  Yes             No              Yes             Yes
Actor isolation:    Table sharding  Process-level   Row-level       Row-level
External service:   No              No              Yes (Neon)      Yes (any PG)
TCP sockets:        No              No              No              Yes (Hyperdrive)
Transaction model:  db.batch()      transactionSync BEGIN/COMMIT    BEGIN/COMMIT
                    (D1 batch)      (local SQLite)  (over WS)      (over TCP)
```

### Key differentiators for Neon specifically

1. **No Hyperdrive required.** The `@neondatabase/serverless` package handles
   HTTP + WebSocket transport natively. No Cloudflare binding configuration
   needed beyond the DSN secret.

2. **Dual transport.** HTTP for reads (lowest overhead), WebSocket for writes
   (transaction support). This is unique to Neon's architecture — other
   PostgreSQL providers require TCP (via Hyperdrive) for everything.

3. **Cold start penalty.** Neon auto-suspends compute after inactivity. First
   query after suspension adds ~200-300ms. This is a Neon-specific trade-off
   not present with D1, DO, or always-on PostgreSQL via Hyperdrive.

4. **Shared SQL layer.** `PgEngineBase` is shared with `HyperdriveEngine`.
   Switching between Neon and Hyperdrive requires zero SQL changes — only the
   transport layer differs.
