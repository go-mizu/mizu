# 0731 — Arctic Publish: Live Progress Tracking

Adds a background heartbeat system to the arctic publish pipeline that:
1. Writes `states.json` to the local repo root every **1 minute**
2. Auto-commits `states.json` + `README.md` to HuggingFace every **5 minutes**
3. Embeds a live "Current Session" section in README.md showing real-time progress

This makes long-running publishes (days to weeks) observable via HF without SSH access,
and provides resumability context if the process is killed.

---

## states.json Schema

Path: `{RepoRoot}/states.json` → committed as `states.json` at HF repo root.

```json
{

  "started_at": "2026-03-15T01:00:00Z",
  "updated_at": "2026-03-15T03:45:12Z",
  "phase": "processing",

    "type": "comments",
    "phase": "processing",

    "bytes_total": 0
```

### Field Descriptions

**Top-level:**

| Field | Type | Description |
|-------|------|-------------|
| `session_id` | string (RFC3339) | Unique ID for this run; set to start time |
| `started_at` | string (RFC3339) | When this `Run()` call started |
| `updated_at` | string (RFC3339) | Last time this file was written to disk |
| `phase` | string | Active phase — see Phase Lifecycle |
| `current` | object\|null | What's being processed right now; null when idle/done |
| `stats` | object | Aggregate counters for this session |

**`current` object:**

| Field | Type | Description |
|-------|------|-------------|
| `ym` | string | Month being processed, e.g. `"2009-06"` |
| `type` | string | `"comments"` or `"submissions"` |
| `phase` | string | Fine-grained phase within this (ym, type) |
| `shard` | int | Current shard index (1-based); 0 when not yet processing |
| `rows` | int64 | Rows written so far in current shard |
| `bytes_done` | int64 | Bytes downloaded so far (download phase only) |
| `bytes_total` | int64 | Total file size in bytes (download phase only) |

**`stats` object:**

| Field | Type | Description |
|-------|------|-------------|
| `committed` | int | (month, type) pairs fully committed this session |
| `skipped` | int | Pairs skipped (already in stats.csv) |
| `total_rows` | int64 | Total rows committed this session |
| `total_bytes` | int64 | Total parquet bytes committed this session |
| `total_months` | int | Total (month, type) pairs in the configured range |

## Phase Lifecycle

```
idle → downloading → processing → committing → idle (next pair)
                                                     ↓ (all done)
                                                    done
```

| Phase | `phase` value | `current` present |
|-------|--------------|-------------------|
| Starting up | `"idle"` | no |
| Fetching .zst | `"downloading"` | yes |
| Stream → chunks → parquet | `"processing"` | yes |
| HF API commit | `"committing"` | yes |
| Between pairs | `"idle"` | no |
| Run finished | `"done"` | no |

The heartbeat writes state in all phases. The 5-minute HF commit fires regardless of phase
(including mid-download and mid-process), so observers always see fresh data.

## Timing Model

```
t=0s      Run() starts → LiveState initialized, initial files written, heartbeat goroutine launched
t=60s     first write tick → states.json + README.md written to disk
t=120s    second write tick → files updated
t=180s    third write tick → files updated
t=300s    first commit tick → check: time since last HF commit
          if > 5min → commitHeartbeat() → HF commit of states.json + README.md
          else → skip (data commit just happened)
```

**Write tick**: every 1 minute — writes `states.json` + `README.md` to local disk only.

**Commit tick**: every 5 minutes — commits to HF only if `≥ 5 minutes since last HF commit`.
This prevents double-committing right after a successful data commit (which also pushes README.md).

**Data commits** (per month/type) also update `lastHFCommit` timestamp, so heartbeat skips
the next commit-tick if one just fired.

**On run completion**: heartbeat goroutine is stopped; a final `writeHeartbeatFiles()` and
`commitHeartbeat()` are called unconditionally to ensure the `done` state is visible on HF.

---

## Concurrency Model

Two goroutines share state:

```
main goroutine          heartbeat goroutine
─────────────────       ──────────────────────────────────
updates LiveState  →→→  reads LiveState (RLock) for snapshot
holds commitMu  ───→→→  waits on commitMu before HF API call
updates lastHFCommit ←← updates lastHFCommit after commit
```

**`LiveState`**: protected by `sync.RWMutex`. Main goroutine calls `ls.Update(fn)` (write lock).
Heartbeat calls `ls.Snapshot()` (read lock) to get a point-in-time copy.

**`commitMu sync.Mutex`**: field on `PublishTask`. Held by both data commits (`processOne`)
and heartbeat commits. Prevents concurrent HF API calls (HF doesn't support concurrent
commits to the same repo/branch).

**`lastHFCommit atomic.Int64`**: stores `time.Now().UnixNano()` after every successful HF
commit (data or heartbeat). Read by heartbeat commit-tick to decide whether to skip.

**Goroutine lifecycle**: heartbeat runs under a derived `context.WithCancel`. The cancel is
called via `defer` in `Run()`, so the goroutine always exits when Run returns.

## Package Changes

### New file: `pkg/arctic/live_state.go`

```go
type LivePhase = string  // "idle"|"downloading"|"processing"|"committing"|"done"

type LiveCurrent struct {
    YM         string    `json:"ym"`
    Type       string    `json:"type"`
    Phase      string    `json:"phase"`
    Shard      int       `json:"shard,omitempty"`
    Rows       int64     `json:"rows,omitempty"`
    BytesDone  int64     `json:"bytes_done,omitempty"`
    BytesTotal int64     `json:"bytes_total,omitempty"`
}

type SessionStats struct {
    Committed   int   `json:"committed"`
    Skipped     int   `json:"skipped"`
    TotalRows   int64 `json:"total_rows"`
    TotalBytes  int64 `json:"total_bytes"`
    TotalMonths int   `json:"total_months"`
}

type StateSnapshot struct {
    SessionID string       `json:"session_id"`
    StartedAt time.Time    `json:"started_at"`
    UpdatedAt time.Time    `json:"updated_at"`
    Phase     string       `json:"phase"`
    Current   *LiveCurrent `json:"current,omitempty"`
    Stats     SessionStats `json:"stats"`
}

type LiveState struct{ mu sync.RWMutex; snap StateSnapshot }

func NewLiveState(totalMonths int) *LiveState
func (s *LiveState) Update(fn func(*StateSnapshot))  // write lock
func (s *LiveState) Snapshot() StateSnapshot          // read lock, deep copy

func WriteStateJSON(cfg Config, snap StateSnapshot) error  // atomic write via temp+rename
```

## Error Handling

| Error | Behavior |
|-------|----------|
| `WriteStateJSON` fails | log to stderr, continue — not fatal |
| `GenerateREADMEWithLive` fails | log to stderr, skip README update this tick |
| `commitHeartbeat` HF API fails | log to stderr, continue — retry next 5-min tick |
| Main data commit fails | propagate error, stops run (existing behavior unchanged) |

Heartbeat failures never crash the run. The worst case is a stale states.json on HF
(up to 5 minutes behind) if the commit fails transiently.

## Commit Message Format

| Commit type | Message |
|-------------|---------|
| Heartbeat (mid-download) | `progress: downloading 2009-06/comments` |
| Heartbeat (mid-process) | `progress: processing 2009-06/comments shard 2` |
| Heartbeat (idle) | `progress: idle (47 committed)` |
| Data commit | `add comments/2009-06 2009/06 (1 shards, 5.1M rows)` (unchanged) |
| Final (done) | `progress: done — 243 committed` |
