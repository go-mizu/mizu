## Goal

Implement `pkg/scrape/ebay` and `cli/ebay.go` following the established scraper pattern used by `pkg/scrape/goodread`, `pkg/scrape/amazon`, and `pkg/scrape/pinterest`.

The first implementation centers on eBay item pages and search-result pages:

- `item` is the primary durable entity
- `search` is the discovery surface that expands the crawl frontier
- all long-running work is organized as `pkg/core.Task[State, Metric]`

This gives a usable "crawl everything from eBay" workflow without inventing a separate crawler architecture:

1. seed one or more search queries
2. crawl queued search pages
3. expand into item pages
4. persist item records plus search snapshots into DuckDB

## Tasks

### `ItemTask`

`pkg/core.Task[ItemState, ItemMetric]`

Responsibilities:

1. fetch item HTML
2. detect interruption/challenge pages
3. parse item metadata from JSON-LD, meta tags, and stable HTML fallbacks
4. upsert the item row
5. discover and enqueue related item URLs
6. mark the queue item done / failed

### `SearchTask`

`pkg/core.Task[SearchState, SearchMetric]`

Responsibilities:

1. build or normalize the search URL
2. fetch up to `Config.MaxPages`
3. parse each search page for result item URLs and pagination
4. store search-page snapshots
5. enqueue discovered item URLs
6. mark the queued search URL done / failed

### `CrawlTask`

`pkg/core.Task[CrawlState, CrawlMetric]`

Responsibilities:

1. pop queue items from `state.duckdb`
2. dispatch by `entity_type`
3. emit periodic progress snapshots
4. stop when the queue is drained

## CLI

Register `search ebay` in `cli/root.go`.

Subcommands:

```text
search ebay item   <id|url>
search ebay search <query>
search ebay seed   --file queries.txt --type search
search ebay seed   --file items.txt   --type item
search ebay crawl  --workers 2
search ebay info
search ebay jobs
search ebay queue  --status pending --limit 20
```

Shared flags:

- `--db`
- `--state`
- `--delay`
- `--max-pages`

`crawl` adds:

- `--workers`

## Parser Strategy

Because eBay item pages can intermittently serve interruption/challenge HTML, the parser should:

1. detect challenge pages explicitly
2. prefer embedded `application/ld+json` product data when available
3. fall back to canonical/meta tags and stable text selectors
4. keep parsing resilient even when some fields are missing

Search pages are simpler and can be scraped from server-rendered `.s-item` cards plus `.pagination__next`.

## First-Pass Limits

This implementation intentionally does not depend on a browser session. It is optimized for the public HTML paths that are reachable without headless automation, and it records failures cleanly when eBay serves an interruption page.
