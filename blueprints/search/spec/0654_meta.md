# 0654 â€” Dashboard Metadata Precache (DuckDB/SQLite + Pluggable Store)

## Goal

Enhance `search cc fts dashboard` so overview/data endpoints load from pre-cached
metadata stored in a local DB (`sqlite` or `duckdb`) instead of walking the full
crawl directory on every request.

Also introduce a Go-stdlib-inspired abstraction so metadata storage backends are
swappable in the future via drivers.

---

## Why

Current behavior:

- `GET /api/overview` and `GET /api/crawl/{id}/data` call `ScanDataDir(...)`.
- `ScanDataDir` recursively walks `warc/`, `markdown/`, `pack/`, and `fts/`.
- On large crawls this is O(number of files) per request and causes slow initial
  dashboard render and repeated latency on refresh/navigation.

Target behavior:

- Read precomputed summary in O(1) DB queries.
- Return cached summary immediately (< 20ms typical local).
- Refresh cache in background when stale.

---

## Scope

### In Scope

- Metadata cache persistence in `sqlite` and `duckdb`.
- Driver abstraction for metadata store (register/open model).
- Cache manager with stale detection + async refresh.
- API response extension with cache status metadata.
- Dashboard startup prewarm for active crawl.
- Integration with job completion to trigger targeted refresh.

### Out of Scope

- Replacing pipeline/index data storage.
- Distributed/shared remote metadata store.
- Full inotify/FSEvents real-time file watching (can be future work).

---

## Requirements

1. Keep existing JSON fields in `DataSummary` backward-compatible.
2. Add cache metadata fields without breaking current frontend.
3. If cache backend fails, transparently fall back to filesystem scan.
4. Support both `sqlite` (default) and `duckdb` backends.
5. Design backend abstraction so new drivers can be added without changing API
   handlers.
6. Never block overview response on a long refresh when cached data exists.

---

## High-Level Design

### Components

1. `MetaManager` (high-level service)
- Reads cached summaries.
- Determines staleness.
- Schedules async refresh (singleflight per crawl).
- Writes refreshed summaries to store.

2. `metastore` package (driver-agnostic facade)
- `Register(name, driver)`
- `Open(driverName, dsn, opts)`
- `Store` API used by `MetaManager`

3. Drivers
- `metastore/drivers/sqlite`
- `metastore/drivers/duckdb`

4. Existing scanner reused as refresh producer
- `ScanDataDir(crawlDir)` remains source of truth for filesystem stats.
- Now used mostly in background refresh or fallback path.

## Go Stdlib-Inspired Abstraction

Inspired by `database/sql` + `database/sql/driver`:

| Stdlib Pattern | Metadata Cache Equivalent |
|---|---|
| `sql.Register(name, driver)` | `metastore.Register(name, driver)` |
| `sql.Open(driver, dsn)` | `metastore.Open(driver, dsn, opts)` |
| blank-import driver package | blank-import metadata driver package |
| app code depends on `*sql.DB` | app code depends on `metastore.Store` |

### Proposed package layout

```text
pkg/index/web/meta/
  manager.go              # stale policy, refresh orchestration
  types.go                # summary/cache status structs

pkg/index/web/metastore/
  registry.go             # Register/Open/List
  store.go                # Store interface + facade
  config.go               # options (ttl, busy_timeout, etc.)

pkg/index/web/metastore/driver/
  driver.go               # low-level driver contracts

pkg/index/web/metastore/drivers/sqlite/
  sqlite.go               # init schema, CRUD

pkg/index/web/metastore/drivers/duckdb/
  duckdb.go               # init schema, CRUD
```

### Core interfaces (spec-level)

```go
// pkg/index/web/metastore/store.go
type Store interface {
    Name() string

```

## Cache Data Model

`DataSummary` remains the canonical payload fields:

- `crawl_id`
- `warc_count`, `warc_total_size`
- `md_shards`, `md_total_size`, `md_doc_estimate`
- `pack_formats`
- `fts_engines`
- `fts_shard_count`

Additional metadata attached to response:

- `meta_backend` (`sqlite` | `duckdb` | `scan-fallback`)
- `meta_generated_at` (RFC3339)
- `meta_stale` (bool)
- `meta_refreshing` (bool)
- `meta_last_error` (string, optional)

### Logical schema (common across backends)

1. `crawl_summary`
- `crawl_id TEXT PRIMARY KEY`
- `warc_count BIGINT NOT NULL`
- `warc_total_size BIGINT NOT NULL`
- `md_shards BIGINT NOT NULL`
- `md_total_size BIGINT NOT NULL`
- `md_doc_estimate BIGINT NOT NULL`
- `generated_at TEXT NOT NULL`
- `scan_duration_ms BIGINT NOT NULL DEFAULT 0`

2. `pack_summary`
- `crawl_id TEXT NOT NULL`
- `format TEXT NOT NULL`
- `bytes BIGINT NOT NULL`
- `PRIMARY KEY (crawl_id, format)`

3. `fts_summary`
- `crawl_id TEXT NOT NULL`
- `engine TEXT NOT NULL`
- `bytes BIGINT NOT NULL`
- `shard_count BIGINT NOT NULL`
- `PRIMARY KEY (crawl_id, engine)`

4. `refresh_state`
- `crawl_id TEXT PRIMARY KEY`
- `status TEXT NOT NULL` (`idle|refreshing|error`)
- `started_at TEXT`
- `finished_at TEXT`
- `last_error TEXT`
- `generation BIGINT NOT NULL DEFAULT 0`

Write strategy:

- On refresh success, replace all rows for a crawl in one transaction:
  - upsert `crawl_summary`
  - delete+insert `pack_summary` for crawl
  - delete+insert `fts_summary` for crawl
  - update `refresh_state`

## Refresh Policy

### Triggers

1. Dashboard startup prewarm for active crawl.
2. Read-time stale check (`meta_refresh_ttl`, default `30s`).
3. Job completion hook (`download`, `pack`, `index`, future `markdown`).
4. Manual refresh endpoint (optional but recommended).

### Rules

1. If cached summary exists:
- Return it immediately.
- If stale and not currently refreshing: trigger async refresh.

2. If cache miss:
- Run synchronous scan once for that crawl.
- Persist result.
- Return scanned summary.

3. Refresh de-dup:
- Single refresh per crawl at a time (in-process singleflight/mutex map).

4. Error handling:
- Keep serving last good snapshot.
- Mark `refresh_state.status=error` and include `meta_last_error`.

## CLI/Config Changes

`search cc fts dashboard` new flags:

- `--meta-driver string` (default `sqlite`, options: `sqlite|duckdb|none`)
- `--meta-dsn string` (optional explicit DB path)
- `--meta-refresh-ttl duration` (default `30s`)
- `--meta-prewarm` (default `true`)

Default paths:

- sqlite: `~/data/common-crawl/.meta/dashboard_meta.sqlite`
- duckdb: `~/data/common-crawl/.meta/dashboard_meta.duckdb`

If `--meta-driver=none`, behavior matches current direct scan mode.

## Performance Targets

For crawl directories with large markdown/fts trees:

1. Warm cached `GET /api/overview` p50: `< 20ms`
2. Warm cached `GET /api/overview` p95: `< 60ms`
3. No repeated full directory walk for each tab switch.
4. First load after startup:
- if prewarmed: same as warm targets
- if cold miss: one-time scan then cached for subsequent calls

## Risks and Mitigations

1. SQL dialect differences (`sqlite` vs `duckdb`)
- Mitigation: keep schema simple and driver-specific SQL isolated.

2. Stale cache after out-of-band file changes
- Mitigation: TTL refresh + manual refresh endpoint + startup prewarm.

3. Driver init failures in user environment
- Mitigation: auto-fallback to scan mode, include backend info in response.

4. Concurrent refresh storms
- Mitigation: per-crawl singleflight lock + refresh_state guard.

---

## Acceptance Criteria

1. Dashboard overview/crawl-data uses cache by default.
2. Metadata persists on disk and survives process restart.
3. Both sqlite and duckdb backends function behind same abstraction.
4. Adding a new backend only requires:
- new driver package implementing `driver.Driver`
- blank import for registration
- no handler logic changes.
5. Existing JSON consumers remain compatible.
