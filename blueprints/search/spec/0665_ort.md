# spec/0665: ORT (ONNX Runtime) Rust Embedder — Research & Benchmarks

## Context

Goal: Replace Go embedding pipeline (358 vec/s llamacpp on M4) with a Rust embedder
targeting 18,000+ vec/s on M4 via Apple Neural Engine (ANE).

### All Approaches Benchmarked (M4 Mac, 10K chunks, real Go/markdown)

| Approach | Backend | Batch | vec/s | P50 ms | RSS MB | Status |
|----------|---------|-------|-------|--------|--------|--------|
| **ORT (Rust)** | **CPU** | **16** | **146** | **116** | **366** | **Best Rust, 2x candle** |
| ORT (Rust) | CoreML | 16 | 55 | — | 2,871 | Dispatch overhead kills it |
| ORT (Rust) | CoreML | 64 | 34 | — | 8,592 | Worse with large batch |
| Candle 0.9.2 | Accelerate CPU | 32 | 60 | 1,902 | 535 | Single-threaded f32 |
| Candle 0.9.2 | Metal GPU | — | N/A | N/A | N/A | **BLOCKED**: no LayerNorm |
| fastembed 5.x | ONNX CPU | 64 | 52 | 1,160 | 2,850 | Huge RSS, slower than ORT |
| **Go llamacpp** | **GGML CPU** | **64** | **358** | — | — | **Overall winner** (Q4/Q8) |
| Go ONNX driver | CPU | — | 117 | — | — | Existing Go pipeline |

**Key findings:**
- Candle 0.9.2 Metal is broken for BERT models (missing LayerNorm kernel,
  tracked in [huggingface/candle#2832](https://github.com/huggingface/candle/issues/2832))
- CPU-only inference (f32, no quantization) tops out at ~60 vec/s regardless of batch size
- llamacpp is fast because of GGML quantized weights (Q4/Q8) + SIMD matmul
- metal-candle 1.3 incompatible with candle 0.9.2 (Metal API mismatch)

### Why ORT + CoreML

The `ort` crate (ONNX Runtime Rust bindings) provides access to Apple's CoreML
execution provider, which can route compute to the Apple Neural Engine (ANE):

```
Rust (ort) → ONNX Runtime → CoreML EP → Core ML framework → ANE
```

**M4 ANE specs:** 19 TFLOPS FP16 at 2.8W — capable of massive throughput for small models.

## ORT Crate Details

### Version & Dependencies

```toml
ort = { version = "=2.0.0-rc.11", features = ["coreml"] }
ndarray = "0.16"
tokenizers = { version = "0.22", default-features = false, features = ["onig"] }
```

- Wraps ONNX Runtime 1.24
- Requires Rust edition 2024 (rustc 1.88+)
- `coreml` feature enables CoreML EP on macOS
- `download-binaries` (default) fetches prebuilt libonnxruntime

### CoreML Execution Provider

```rust
use ort::execution_providers as ep;

let session = Session::builder()?
    .with_execution_providers([
        ep::CoreMLExecutionProvider::default()
            .with_subgraphs()                      // CoreML on subgraphs too
            .with_compute_units(ComputeUnits::All)  // CPU + GPU + ANE
            .build(),
    ])?
    .with_optimization_level(GraphOptimizationLevel::Level3)?
    .commit_from_file("model.onnx")?;
```

**Configuration options:**
- `with_subgraphs()` — enables CoreML for control-flow subgraphs
- `with_compute_units(...)` — `CPUOnly`, `CPUAndGPU`, `CPUAndNeuralEngine`, `All`
- First load compiles model for CoreML (can take seconds), cached after

### Model Files

From `sentence-transformers/all-MiniLM-L6-v2` on HuggingFace:

| File | Size | Use Case |
|------|------|----------|
| `onnx/model.onnx` | 90.4 MB | Full precision (best for CoreML FP16) |
| `onnx/model_O2.onnx` | 90.3 MB | Optimized level 2 |
| `onnx/model_O4.onnx` | 45.2 MB | Quantized |
| `onnx/model_qint8_arm64.onnx` | 23 MB | INT8 ARM64 (ANE dequants to FP16) |
| `tokenizer.json` | 466 KB | WordPiece tokenizer |

**Note:** ANE dequantizes INT8 to FP16 before compute, so INT8 saves only memory
bandwidth, not compute. Full-precision model may be optimal for CoreML.

### Inference Pipeline

```rust
// 1. Tokenize batch
let encodings = tokenizer.encode_batch(texts, true)?;
let ids: Vec<i64> = encodings.iter().flat_map(|e| e.get_ids().map(|i| i as i64)).collect();
let mask: Vec<i64> = encodings.iter().flat_map(|e| e.get_attention_mask().map(|i| i as i64)).collect();

// 2. Create tensors [batch_size, seq_len]
let t_ids = TensorRef::from_array_view(([batch, seq_len], &*ids))?;
let t_mask = TensorRef::from_array_view(([batch, seq_len], &*mask))?;

// 3. Run inference
let outputs = session.run(ort::inputs![t_ids, t_mask]?)?;

// 4. Extract embeddings — output[1] is sentence_embedding (already pooled)
let embeddings = outputs[1].try_extract_array::<f32>()?.into_dimensionality::<Ix2>()?;

// 5. L2 normalize
```

The HuggingFace ONNX export of all-MiniLM-L6-v2 includes mean pooling in the graph,
so `outputs[1]` is the pooled sentence embedding `[batch, 384]`.

### Output Format (Compatible with Go Pipeline)

Same format as `search cc fts embed`:

```
output/
  vectors.bin    # Raw float32, vector i at offset i × dim × 4
  meta.jsonl     # {"id":"file:idx", "file":"...", "chunk_idx":N, "text_len":N, "dim":384}
  stats.json     # Summary statistics
```

## Actual Benchmark Results (March 2026)

### ORT CPU Batch Size Sweep (100 markdown files, 1036 chunks)

| Batch Size | vec/s | Notes |
|-----------|-------|-------|
| 16 | **146** | Best — less padding waste |
| 32 | 137 | |
| 64 | 133 | |
| 128 | 134 | |
| 256 | 136 | |

Smaller batches win because `PaddingStrategy::BatchLongest` pads all inputs to the
longest in the batch — with batch=16, less wasted compute on short sequences.

### CoreML Is a Bust

| Config | vec/s | RSS MB | Why |
|--------|-------|--------|-----|
| CoreML batch=16 | 55 | 2,871 | XPC dispatch overhead |
| CoreML batch=64 | 34 | 8,592 | CoreML model compilation + memory |
| CPU batch=16 | 146 | 366 | No overhead, just inference |

CoreML adds 2-4x overhead for small models (22M params). The XPC IPC between the
app and the CoreML daemon dominates. ANE never gets to shine because the model is
too small. `Context leak detected, msgtracer returned -1` also indicates issues.

### Why llamacpp Wins

llamacpp uses GGML quantized weights (Q4_K_M, Q8_0) which:
- Reduce matmul compute by 4-8x vs f32
- Use hand-tuned ARM NEON / AVX2 kernels
- Keep model in L2 cache (23MB Q4 vs 90MB f32)

For embedding models on CPU, **quantization is the #1 optimization**, not GPU offload.

## Expected Performance

### M4 Mac (CPU vs CoreML)

| Config | Expected vec/s | Reasoning |
|--------|---------------|-----------|
| CPU, batch=64 | ~400-800 | ORT is more optimized than candle for CPU |
| CoreML (GPU), batch=64 | ~2,000-5,000 | Metal GPU via CoreML |
| CoreML (ANE), batch=64 | ~5,000-20,000 | Neural Engine at 19 TFLOPS |
| CoreML (ANE), batch=256 | ~10,000-22,000 | Larger batches amortize overhead |

**Caveat:** CoreML adds 2-4x overhead for small models due to XPC dispatch.
For a 22M-param model like MiniLM, overhead may dominate at small batch sizes.

### Server2 (EPYC 6 vCPU, no GPU)

| Config | Expected vec/s |
|--------|---------------|
| CPU, batch=64 | ~100-200 |
| CPU + AVX2 quantized | ~200-400 |

## Key Risks

1. **CoreML compilation overhead**: First load can take 5-10s as CoreML compiles the model
2. **Operator coverage**: Not all ONNX ops supported by CoreML EP; unsupported ops fall back to CPU
3. **ANE routing**: CoreML decides which ops go to ANE vs CPU — not controllable
4. **Batch size sensitivity**: ANE may not benefit from very large batches if model is too small
5. **ORT version pinning**: `=2.0.0-rc.11` is a release candidate, API may change

## Go ONNX Driver Improvements

Two fixes applied to `pkg/embed/driver/onnx/onnx.go`:

1. **Session reuse**: Create session once in `Open()`, reuse in `Embed()`. Previously
   created a new session per batch (massive overhead).
2. **Mutex for thread safety**: ONNX `session.Run()` is not thread-safe. Added `sync.Mutex`
   to serialize concurrent embed workers.
3. **Padding instead of resizing**: Pad last batch to session capacity instead of
   destroying/recreating session (which crashed with SIGABRT from concurrent goroutines).

### CoreML on Go

**Not possible** with Homebrew onnxruntime — the bottle doesn't include CoreML EP.
Would need a custom build from source with `-Donnxruntime_USE_COREML=ON`.
The Rust `ort` crate bundles its own prebuilt ORT with CoreML support.

### Go ONNX Bottleneck: Tokenizer

The pure-Go WordPiece tokenizer always pads to `maxSeqLen=128`, wasting compute on
short texts. The Rust HuggingFace `tokenizers` crate uses `BatchLongest` padding
(pad to longest in batch), which is ~2x more efficient for typical 50-80 token inputs.
