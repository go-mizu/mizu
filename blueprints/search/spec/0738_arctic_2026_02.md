# 0738: Process Arctic 2026-02 on Server2

## Problem

The main arctic pipeline (`arctic_run12`) is processing months sequentially from 2005-12.
It's currently on 2011-xx and won't reach 2026-02 for days/weeks. We need 2026-02
committed to HuggingFace now.

## File Sizes (from zst_sizes.json)

| File | Size | Est. Rows |
|------|------|-----------|
| `comments/RC_2026-02.zst` | 39.6 GB (torrent) / 42.5 GB (catalog) | ~350M |
| `submissions/RS_2026-02.zst` | 17.8 GB | ~80M |
| **Total** | **~57 GB** | **~430M** |

## Server2 Constraints

| Resource | Total | Currently Used | Available |
|----------|-------|----------------|-----------|
| RAM | 11 GB | 8.8 GB | 2.8 GB |
| Disk | 193 GB | 74 GB | 119 GB |
| CPU | 6 cores | ~400% | ~200% |

### Why We Can't Run In Parallel With arctic_run12

1. **stats.csv corruption**: Both processes read/write `$HOME/data/arctic/repo/stats.csv`.
   `WriteStatsCSV` does atomic rename, but concurrent read-modify-write from two processes
   causes lost rows (process A reads, process B writes, process A overwrites B's changes).

2. **HF commit conflicts**: Both commit `stats.csv` to the same HF repo path. Each commit
   replaces the file. A 2026-02 commit with only its rows would erase all 2005-2011 entries.

3. **RAM**: arctic_run12 uses 2.9 GB RSS. The 2026-02 zstd decoder alone needs 2 GB.
   Together: 5 GB + DuckDB (512 MB) + OS → OOM risk on 11 GB.

4. **Disk**: 42.5 GB comments zst + 17.8 GB submissions zst = 60.3 GB. Plus arctic_run12's
   in-progress downloads (3.9 GB). With min-free-gb=30, need 90+ GB free. Barely fits.

## Plan

### Step 1: Stop arctic_run12

Safe because stats.csv tracks committed months. On restart, it skips all committed months
and resumes from where it left off.

```bash
# Find and kill the arctic process
screen -S arctic_run -X stuff $'\003'  # Send Ctrl+C for graceful shutdown
# Wait 10s for graceful shutdown, then verify
sleep 10 && ps aux | grep "arctic publish" | grep -v grep
```

### Step 2: Clean Up Disk

Remove arctic_run12's leftover .zst files and work directory to maximize free space.

```bash
rm -f /root/data/arctic/raw/*.zst
rm -rf /root/data/arctic/work/*
df -h /  # Should show ~123 GB free
```

### Step 3: Launch 2026-02 Session

```bash
screen -dmS arctic_2026_02 bash -c '
  exec >> /root/log/arctic_2026_02.log 2>&1
  echo "[$(date)] starting 2026-02 processing"
  export PATH=/root/bin:$PATH
  GOMEMLIMIT=8GiB HF_TOKEN=$(cat /root/.hf_token) \
    /root/bin/search-linux-noble arctic publish \
      --from 2026-02 --to 2026-02 \
      --min-free-gb 20 \
      --max-commit-stall 120m \
  echo "[$(date)] 2026-02 exited $?"
'
```

Key flags:
- `--from 2026-02 --to 2026-02`: Only process this month
- `--min-free-gb 20`: Lower than default 30 since we're tight on disk
- `--max-commit-stall 120m`: Higher stall timeout — 42.5 GB download + processing
  will take 1-2 hours. Default 45m would kill the pipeline during slow torrent download.
- `GOMEMLIMIT=8GiB`: Same as arctic_run12

### Step 4: Monitor

```bash
# Watch log
tail -f /root/log/arctic_2026_02.log

# Check progress
grep -E "(downloading|processing|committed|SKIP|Error)" /root/log/arctic_2026_02.log

# Check disk
df -h /

# Check memory
free -h && ps aux --sort=-rss | head -5
```

## Risk Analysis

### OOM Risk: LOW

With arctic_run12 stopped, ~5-6 GB available for the 2026-02 process:
- zstd decoder: 2 GB (semaphore ensures only 1 at a time)
- DuckDB: 512 MB per instance
- Scanner + buffers: ~100 MB
- Go runtime: ~500 MB
- **Total**: ~3.1 GB — fits within 5-6 GB available

### Disk Full Risk: LOW-MEDIUM

Timeline with arctic_run12 stopped and work cleaned:
1. Download comments (42.5 GB) → ~80 GB free
2. Process comments (chunks streamed, .zst deleted after) → ~95 GB free after .zst delete
3. Parquet shards accumulate (~15 GB for 350M rows) → ~80 GB free
4. Upload shards → cleaned up → ~95 GB free
5. Download submissions (17.8 GB) → ~77 GB free
6. Process + upload → done

Minimum free during pipeline: ~65 GB (during comments download + submissions download overlap).
Well above min-free-gb=20.

### Download Stall Risk: MEDIUM

2026-02 files are recent — BitTorrent seeders likely healthy (unlike 2011-xx months which
had stalling peers). But 42.5 GB at even 5 MB/s takes 2.3 hours.

Mitigations already in code:
- 3-min stall timeout per download attempt
- 5 retry attempts with backoff
- Non-fatal skip on failure (new refactored code)

If comments download fails, submissions can still proceed.

### Rate Limit Risk: LOW

Only 2 HF commits (comments + submissions). Rate limit retries handle 429s with
server-requested Retry-After + 30s buffer.

### Crash Risk: LOW

- SIGKILL recovery: stats.csv only updated after successful HF commit
- Connection reset handling: 15s wait + verify ops exist on HF
- Non-fatal download/process failures (refactored code)

## Expected Timeline

| Phase | Duration | Notes |
|-------|----------|-------|
| Comments download | 1-3 hours | 42.5 GB via torrent, depends on seeder speed |
| Comments process | 30-60 min | ~350M rows at ~100K rows/s, 700+ shards |
| Comments upload | 10-20 min | ~15 GB parquet, batched 50 ops at a time |
| Submissions download | 30-90 min | 17.8 GB via torrent |
| Submissions process | 15-30 min | ~80M rows |
| Submissions upload | 5-10 min | ~5 GB parquet |
| **Total** | **3-7 hours** | Depends heavily on torrent speed |

## After Completion

Restart arctic_run12 to continue processing remaining months:

```bash
screen -dmS arctic_run bash -c '
  export PATH=/root/bin:$PATH
  GOMEMLIMIT=8GiB HF_TOKEN=$(cat /root/.hf_token) \
    /root/bin/search-linux-noble arctic publish \
      --from 2005-12 --max-commit-stall 45m \
    2>&1 | tee /root/log/arctic_run13.log
  exec bash
'
```

It will skip all already-committed months (including 2026-02) and resume from
the first uncommitted month.
