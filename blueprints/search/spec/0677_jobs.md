# 0677 Jobs Rearchitecture

## Problem

1. **No persistence**: Jobs are fully in-memory (`map[string]*Job`). Server restart loses all history.
2. **No cleanup**: Jobs accumulate forever. No clear/delete functionality.
3. **"syncing" bug**: `handleOverview` hardcodes `resp.Meta.Stale = true` (line 810), causing the frontend watchdog to trigger infinite refresh loops. Header chip always shows "syncing".
4. **Poor listing UX**: Flat table with no grouping, no individual job page, no pagination.
5. **No job detail page**: No route for `/jobs/{id}`.

## Solution

### Fix 1: "syncing" bug (DONE)

- Add `IsStale()` to MetaManager using in-memory `lastRefreshAt` map (set on successful refresh).
- Replace `resp.Meta.Stale = true` with `resp.Meta.Stale = s.Meta.IsStale(s.CrawlID)`.
- Only trigger refresh when actually stale AND not already refreshing.
- Remove frontend redundant `apiMetaRefresh()` call in `refreshDashboardContext()`.

### Fix 2: Job persistence via metastore

Add a `jobs` table to the existing metastore schema:

```sql
CREATE TABLE IF NOT EXISTS jobs (
  id TEXT PRIMARY KEY,
  type TEXT NOT NULL,
  status TEXT NOT NULL,
  config_json TEXT NOT NULL,
  progress REAL NOT NULL DEFAULT 0,
  message TEXT NOT NULL DEFAULT '',
  rate REAL NOT NULL DEFAULT 0,
  error TEXT NOT NULL DEFAULT '',
  started_at TEXT NOT NULL,
  ended_at TEXT
);
```

Extend `metastore.Store` interface with:
- `ListJobs(ctx, limit, offset) ([]JobRecord, error)`
- `PutJob(ctx, JobRecord) error`
- `DeleteJobs(ctx) error` (clear all)

`JobManager` changes:
- Accept optional `metastore.Store` reference.
- On Create/Complete/Fail/Cancel/UpdateProgress: persist to store (async, non-blocking).
- On startup: load completed/failed/cancelled jobs from store into `jobs` map.
- Running/queued jobs are NOT restored (they can't resume after restart).
- Add `Clear()` method that deletes all non-active jobs from memory + store.

### Fix 3: Enhanced job listing frontend

**Job grouping**: Group by `type` (download, markdown, index) with collapsible sections.

**Summary cards**: Total, Running, Completed, Failed (keep existing 4-card layout but improve).

**Single job page** (`#/jobs/{id}`):
- Full job detail: config, progress, timing, error message
- Cancel button for active jobs
- Progress bar with rate

**Clear button**: "Clear History" removes all completed/failed/cancelled jobs.

**Pagination**: Simple prev/next for job history (most sessions have <100 jobs, full pagination unnecessary).
