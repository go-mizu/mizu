## Part 1: SerpAPI Automation (PAUSED)

### Status: On hold — captcha bypass too fragile

Built in `pkg/serp/`:
- `serpapi.go` — SerpAPI client (register, verify, search, account)
- `register.go` — Registration orchestrator
- `register_rod.go` — Rod browser fallback for captcha bypass
- `captcha_bypass.go` — Captcha solving attempts
- `mailtm.go` — Disposable email via mail.tm
- `store.go` — API key storage
- `cli/serp.go` — CLI commands (register, search, list, status)

### SerpAPI Details (for reference)

- Free tier: 100 searches/month/account
- Registration: Rails web app, `POST https://serpapi.com/users` with JSON body
- Verification: email link `https://serpapi.com/users/confirmation?confirmation_token=...`
- Login: `POST https://serpapi.com/users/sign_in.json` → returns `api_key`
- Search: `GET https://serpapi.com/search.json?q=<query>&api_key=<key>&engine=google`
- Account: `GET https://serpapi.com/account.json?api_key=<key>` → `plan_searches_left`

### mail.tm API

- `GET https://api.mail.tm/domains` → list domains
- `POST /accounts` → create `{"address":"user@domain","password":"pass"}`
- `POST /token` → get Bearer token
- `GET /messages` → poll for verification email (up to 90s, 3s interval)

### Rod Fallback

Navigate to `https://serpapi.com/users/sign_up`, fill form, submit. Falls back when HTTP registration returns 4xx.

### Key Storage

File: `$HOME/data/serp/keys.json` — array of `{email, password, api_key, registered_at, searches_left, last_checked}`

### CLI Commands (existing)

```
search serp register [--count N]
search serp search <query>
search serp list
search serp status
```

## Part 2: Free Tier SERP API Alternatives (NEW)

### Problem

SerpAPI auto-registration requires captcha bypass which is fragile. Pivot to APIs with simple API key signup (manual, one-time) and generous free tiers.

### Research Results

#### 1. Serper.dev — BEST FREE TIER ⭐

- **Free**: 2,500 queries (one-time grant, no credit card)
- **Speed**: 1-2 seconds
- **API**: `POST https://google.serper.dev/search`
  - Header: `X-API-KEY: <key>`
  - Body: `{"q": "query", "num": 10}`
- **Response**: JSON with organic_results, knowledge_graph, related_searches
- **Signup**: email/password at serper.dev → API key in dashboard
- **Paid**: ~$50/mo for 50K

#### 2. Zenserp.com

- **Free**: 50 searches/month (recurring, no credit card)
- **API**: `GET https://app.zenserp.com/api/v2/search?apikey=KEY&q=QUERY`
- **Response**: JSON with organic_results, related_searches
- **Signup**: simple registration
- **Paid**: $49.99/mo (25K) → $899.99/mo (1M)
- **Note**: Failed requests don't count

#### 3. SerpStack.com

- **Free**: 100 searches/month
- **API**: `GET http://api.serpstack.com/search?access_key=KEY&query=QUERY`
- **Response**: JSON with organic_results, ads, knowledge_graph
- **Paid**: starts $29.99/mo
- **Caveat**: Free tier = HTTP only (no HTTPS)

#### 4. SearchAPI.io

- **Free**: 100 searches (one-time, no credit card)
- **API**: `GET https://www.searchapi.io/api/v1/search?api_key=KEY&q=QUERY`
- **Response**: JSON with organic_results, answer_box
- **Paid**: $40/mo (10K) → $500/mo (250K)

#### 5. Serply.io

- **Free**: Monthly credits (exact count unclear)
- **API**: `GET https://api.serply.io/v1/search/q=QUERY`
  - Header: `X-Api-Key: <key>`
- **Signup**: simple, API key in dashboard
- **Paid**: higher usage tiers

### Priority for Implementation

1. **Serper.dev** — 2,500 free, fast, simple API
2. **SearchAPI.io** — 100 free, clean API
3. **Zenserp.com** — 50/month recurring

### Implementation Plan

1. Add `SerpProvider` interface in `pkg/serp/`:
   ```go
   type SerpProvider interface {
       Search(query string, opts SearchOpts) ([]SearchResult, error)
       Name() string
   }
   ```

2. Implement providers: `serper.go` (Serper.dev), `searchapi.go`, `zenserp.go`

3. CLI changes:
   ```
   search serp add-key <provider> <api-key>   # manually add API key
   search serp search -p serper "query"        # search with specific provider
   search serp search "query"                  # auto-pick provider with remaining quota
   ```

4. Key storage: extend `keys.json` with `provider` field

5. Fallback chain: try providers in priority order if one fails/exhausted
