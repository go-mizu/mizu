## High-Level Summary
`qmd` is a local markdown retrieval system with three layers:
1. Index + storage (SQLite + FTS5 + sqlite-vec)
2. Search orchestration (BM25, vector, query expansion, RRF fusion, reranking)
3. Interfaces (CLI, MCP stdio server, MCP HTTP server/daemon)

Its distinctive feature is not just indexing markdown files, but combining:
- per-collection configuration + path-scoped human context strings
- content-addressable storage (`content` table keyed by SHA-256 hash)
- chunked embeddings for semantic retrieval
- typed structured queries (`lex`, `vec`, `hyde`)
- LLM-assisted query expansion + reranking

## Data Model and Schema (from `src/store.ts`)
### Core tables
- `content(hash PRIMARY KEY, doc, created_at)`
  - content-addressable body storage; deduplicates identical file contents across documents
- `documents(id, collection, path, title, hash, created_at, modified_at, active)`
  - maps virtual file paths to content hash
  - soft-deletes via `active = 0`
- `llm_cache(hash, result, created_at)`
  - caches LLM API/model outputs (query expansion/rerank)
- `content_vectors(hash, seq, pos, model, embedded_at, ...)`
  - chunk metadata for embeddings
- `vectors_vec` (sqlite-vec virtual table)
  - actual vector storage/search (cosine distance)
- `documents_fts` (FTS5 virtual table)
  - indexes `filepath`, `title`, `body`

### Important schema behaviors
- FTS table is kept in sync with triggers on `documents` insert/update/delete.
- QMD drops older legacy tables (`path_contexts`, `collections`) and uses YAML config instead.
- Vector table schema is rebuilt if dimensions or options drift.

## Indexing / Update Pipeline
### Collections and config
Collections live in YAML (`~/.config/qmd/<index>.yml`) and include:
- `path`
- `pattern` (glob, default `**/*.md`)
- `context` map (path prefix -> context string)
- `update` command (optional shell command)
- `includeByDefault` (optional)

### `update`
`qmd update` performs:
1. load collection config
2. optionally run per-collection update command (`git pull`, etc.)
3. scan files matching collection glob
4. compute content hash and title
5. insert new content rows / update document mappings
6. deactivate missing documents
7. leave embeddings stale if content changed (to be refreshed by `embed`)

### `embed`
`qmd embed`:
- finds active documents without embeddings
- chunks documents (smart markdown-aware chunking)
- formats document text for embedding prompt template
- generates embeddings via `node-llama-cpp`
- stores per-chunk embeddings and metadata

## Smart Chunking (Key Implementation Detail)
QMD chunking is a notable quality/perf feature in `store.ts`:
- target chunk size: ~900 tokens (char approximation fallback ~3600 chars)
- overlap: ~15%
- prefers semantic breakpoints (headings, code fences, blank lines, list items)
- avoids splitting inside fenced code blocks
- chooses cutoffs using a scored breakpoint system with distance decay

This is important because downstream reranking happens on selected chunks, not full docs.

## Search Backends and Scoring
### 1) FTS / BM25 (`search`)
`searchFTS()` in `store.ts`:
- parses lex query syntax into FTS5 query
- supports:
  - quoted phrases (`"..."`)
  - negation (`-term`, `-"phrase"`)
  - prefix matching for plain terms (`"term"*`)
- uses SQLite `bm25()` and normalizes score to ~`[0,1)` via `abs(score)/(1+abs(score))`

### 2) Vector search (`vsearch`)
`searchVec()`:
- embeds query
- queries `vectors_vec` (sqlite-vec) by cosine distance
- fetches chunk/document metadata via second query (workaround for sqlite-vec JOIN hangs)
- deduplicates by filepath, keeping best chunk
- returns cosine similarity (`1 - distance`)

### 3) Hybrid search (`query`)
`hybridQuery()` pipeline in `store.ts`:
1. BM25 probe for strong exact-signal detection
2. optional LLM query expansion (typed variants `lex`, `vec`, `hyde`)
3. routed retrieval (FTS for `lex`, vector for `vec/hyde`)
4. Reciprocal Rank Fusion (RRF)
5. top-rank bonus (+0.05 for rank1, +0.02 for rank2-3)
6. candidate truncation
7. best-chunk selection per document
8. LLM reranking on chunks
9. position-aware blending (protects top retrieval hits)
10. dedupe + threshold + final top-K

## Structured Query Mode (Important for Agents)
QMD supports a structured multi-line query document format (validated by CLI and used in MCP):
- `lex: ...`
- `vec: ...`
- `hyde: ...`

`structuredSearch()` executes these typed subqueries directly, skipping internal query expansion. This is the preferred path when an external LLM/agent can generate better subqueries.

## Context System (Why QMD is Useful for Agents)
QMD stores human-authored context summaries at:
- global scope (`/`)
- collection root
- arbitrary path prefixes inside collection

At query time, matched documents include the most specific matching context. This improves LLM selection and answer quality when many files are semantically similar.

## Porting Implications for Go (`pkg/qlocal`)
### Easy to port (SQLite + app logic)
- collection/context config model
- content/document schema and FTS triggers
- `get` / `multi-get` / `ls`
- BM25 lex parsing
- RRF fusion and chunk heuristics
- output formatting modes

### Hard to port (requires extra subsystem work)
- `node-llama-cpp` model lifecycle and GGUF management
- sqlite-vec compatibility (or a Go-native vector index alternative)
- full MCP server parity (stdio + Streamable HTTP + daemon semantics)
- high-fidelity query expansion/reranking behavior

## Summary for `pkg/qlocal` Work
QMD is best treated as three separable things:
1. a strong local markdown indexer/retriever (portable now)
2. an LLM-enhanced retrieval pipeline (portable with backend integration)
3. an MCP integration surface (separate deliverable)

A practical Go port should ship in stages, with explicit parity tracking for:
- core retrieval (FTS/get/multi-get/context/status/update)
- vector/hybrid (initial heuristic embeddings, then true LLM embeddings/rerank)
- MCP and model management parity
