# Spec 0602: Achieve 1k pages/s on Remote Server

## Status
- **Target:** 1000+ pages/s crawling throughput on remote server.
- **Environment:** Ubuntu 20.04, < 8GB RAM.
- **Current Status:** Optimized engine locally hit 1,265 pages/s peak. Deployment in progress.

## Root Cause Analysis (Local)
1. **Sequential Probing:** Fixed by parallelizing chunk probing.
2. **URL Serialization:** Fixed by interleaving URLs across domains.
3. **Conservative Defaults:** Increased `MaxConnsPerDomain` to 32 and `DomainFailThreshold` to 20.
4. **Accounting Error:** Speed calculation now strictly counts network-fetched pages.

## Remote Considerations (< 8GB RAM)
- **Worker Memory:** 5,000 workers with sharded HTTP clients and `InsecureSkipVerify: true` (avoiding CGO threads) should fit in ~2-4GB RAM for `status-only` mode.
- **DuckDB Sharding:** 16 shards for results minimizes lock contention and memory spikes during heavy writes.
- **DNS Cache:** Pre-resolution avoids runtime lookup overhead and memory churn.

## Steps to Success
1. **Deploy:** `make deploy-linux` with the enhanced non-interactive script.
2. **Benchmark Remote:** Run `search cc recrawl --file p:0 --limit 100000 --status-only` on the server.
3. **Monitor Resources:** Use `top` or `free -m` (simulated via plan) to ensure memory stays within 8GB.
4. **Tune if needed:** If RAM is tight, reduce `--workers` to 2000-3000 (still should hit 1k/s if latency is low).
