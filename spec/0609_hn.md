# Spec 0609: Hacker News Dataset Import + `search hn`

## Goal

Implement a new `search hn` workflow that can:

1. Discover a fast bulk source for all Hacker News items (prefer parquet over API)
2. Download data to `$HOME/data/hn/` with resume support
3. Import into DuckDB at `$HOME/data/hn/hn.duckdb`
4. Fall back to the official HN Firebase API if the parquet source is unavailable
5. Provide good CLI DX (`list`, `download`, `import`, `status`, `sync`)

## Primary Data Source (Latest, 2026)

### Selected source

Use the ClickHouse SQL playground live Hacker News table:
- Endpoint: `https://sql-clickhouse.clickhouse.com`
- Table: `hackernews.hackernews`
- Export method: chunked `SELECT ... FORMAT Parquet` by `id` ranges

Notes:
- This source is current as of runtime (querying the live ClickHouse `hackernews.hackernews` table).
- API fallback remains available and is also used for a small post-ClickHouse delta catch-up.

### Static snapshot fallback (legacy)

Keep the ClickHouse S3 snapshot as a fallback:
- `https://datasets-documentation.s3.eu-west-3.amazonaws.com/hackernews/2023-08-18.parquet`

### Why this source

- Latest data source via live ClickHouse table (not capped to an older static snapshot)
- Parquet export per `id` chunk from ClickHouse keeps import fast and resumable
- Avoids the very slow item-by-item Hacker News API path for normal usage

### Runtime discovery behavior

`search hn list` and `search hn download` query ClickHouse for:
- `count()`
- `max(id)`
- `max(time)`

`search hn list` also shows static snapshot fallback metadata via `HEAD`.

## Fallback Source (HN API)

### Endpoint

Official Firebase API base:
- `https://hacker-news.firebaseio.com/v0`

Used endpoints:
- `/maxitem.json`
- `/item/<id>.json`

### Fallback strategy

If parquet download is requested in `auto` mode and the source is unreachable / HEAD fails / GET fails, fall back to API mode.

API mode is implemented as **chunked JSONL downloads** for resumability:
- Output dir: `$HOME/data/hn/raw/api/chunks/`
- Chunk file pattern: `items_000000001_000001000.jsonl`
- Completed chunk files are treated as checkpoints
- Partial chunk downloads write to `*.tmp` then atomically rename on success

This gives resume support without complex per-item checkpointing or duplicate tracking.

## Local Storage Layout

Base dir: `$HOME/data/hn/`

Planned layout:

```text
$HOME/data/hn/
  raw/
    items.parquet                  # legacy static snapshot fallback (optional)
    clickhouse/
      id_000000001_001000000.parquet
      ...
    api/
      chunks/
        items_000000001_000001000.jsonl
        ...
  state/
    parquet_head.json             # optional cached HEAD metadata (best effort)
  hn.duckdb                       # imported database
```

## CLI UX

### Root command

`search hn`

### Subcommands

#### `search hn list`
Show:
- remote ClickHouse source metadata (`count`, `max(id)`, `max(time)`)
- static snapshot fallback metadata (size, last-modified) when reachable
- local parquet status (exists/size)
- local API chunk status (count, bytes)
- local DuckDB status (exists/size and row count if present)

#### `search hn download`
Download dataset to local disk.

Flags:
- `--source auto|clickhouse|parquet|api` (default `auto`)
- `--force` (restart existing target)
- `--parallel` (ClickHouse mode chunk download concurrency; errgroup limit)
- `--chunk-id-span` (ClickHouse mode, id range per parquet chunk)
- `--workers` (API mode concurrent requests; default ~32)
- `--chunk-size` (API mode, default 1000)
- `--from-id` / `--to-id` (API mode range override; useful for tests and partial sync)
- `--no-fallback` (fail if parquet source fails instead of using API)

Behavior:
- `auto`: try ClickHouse latest export first, then API fallback (and optional API delta after ClickHouse)
- `clickhouse`: chunked Parquet export from `hackernews.hackernews` (resumable by chunk file)
- `parquet`: resumable HTTP range download to `raw/items.parquet`
- `api`: chunked JSONL download (resumable by chunk files)

#### `search hn import`
Import locally downloaded data into DuckDB.

Flags:
- `--source auto|clickhouse|hybrid|parquet|api` (default `auto`)
- `--db <path>` (default `$HOME/data/hn/hn.duckdb`)
- `--rebuild` (drop/recreate `items` table)

Behavior:
- `auto`: prefer local ClickHouse chunks + API delta (hybrid), otherwise ClickHouse-only, then legacy parquet, then API
- `hybrid`: import ClickHouse chunks, then overlay API delta rows by `id`
- Parquet import: `CREATE TABLE items AS SELECT ... FROM read_parquet(...)`
- API import: `CREATE TABLE items AS SELECT ... FROM read_json_auto(glob)`
- ClickHouse import: `CREATE TABLE items AS SELECT ... FROM read_parquet(clickhouse/*.parquet)` with enum normalization
- Create a `meta` table with import provenance and timestamps
- Add useful indexes (e.g., `id`, `type`, `time`, `by`, `parent`)

#### `search hn status`
Quick local status summary only (no remote calls unless explicitly requested later).

#### `search hn sync`
One-shot wrapper for `download` then `import` with the same source selection flags.

## Import Schema (DuckDB)

### Table: `items`

Store core Hacker News item fields from source plus a derived timestamp:
- `time_ts` = `epoch_ms(CAST(time AS BIGINT) * 1000)` when available
Import query should be tolerant of null/missing fields (`try_cast`).
`words` (ClickHouse derived token array) is intentionally excluded to reduce local DuckDB size and disk pressure.

### Table: `meta`

Key-value metadata for provenance and operational state, e.g.:
- `source_kind` (`parquet` or `api`)
- `source_path`
- `imported_at`
- `row_count`

### Indexes

Attempt (best effort):
- `items(id)`
- `items(type)`
- `items(time)`
- `items(by)`
- `items(parent)`

## Resumability Details

### ClickHouse chunk download resume (primary)

- Query ClickHouse for `max(id)` and `count()`
- Export `id` ranges to `raw/clickhouse/id_<start>_<end>.parquet`
- Skip existing non-empty chunk files
- Parallelize downloads via `errgroup` with `SetLimit(--parallel)`
- Show per-chunk bytes, chunk speed, and aggregate throughput

### Parquet download resume (legacy fallback)

- `HEAD` remote file to get total size / range support
- If local file exists and `< remote_size`, resume using `Range: bytes=<local_size>-`
- If local size == remote size, treat as complete
- If local size > remote size, require `--force` or fail with a clear error
- Write progress to stdout (bytes, speed, ETA)

### API chunk resume

- Skip chunk files that already exist and are non-empty
- Download remaining chunks with worker pool
- Write chunk to temp file and rename on successful completion
- Each chunk is an independent checkpoint

## Implementation Notes / Tradeoffs

- Primary path is optimized for the user’s request (bulk parquet); API fallback prioritizes correctness + resumability over absolute speed.
- API fallback uses chunked JSONL files rather than a single append-only file to avoid duplicate/resume bookkeeping complexity.
- `list` may do a remote `HEAD`; `status` is local-only for fast repeated checks.
- Hidden env overrides will be supported for testing and development:
  - `MIZU_HN_DATA_DIR`
  - `MIZU_HN_PARQUET_URL`
  - `MIZU_HN_API_BASE_URL`

## Success Criteria

- `search hn list` shows remote and local status
- `search hn download` can resume an interrupted parquet download
- `search hn download --source api` downloads chunked JSONL and resumes by skipping completed chunks
- `search hn import` imports local data into `$HOME/data/hn/hn.duckdb`
- `search hn sync` performs download + import in one step
- Tests pass for package logic and CLI command wiring/workflows
